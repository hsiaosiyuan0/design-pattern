# 人物与势力设定

这份文档用来稳住《乱世里的设计模式》的历史结构，让 23 回不只是“同题材短文”，而是真正共用一条叙事骨架。

## 主角

### 沈策

- 出身寒门，少年时在县学读书，精于兵书与簿牍
- 初入河东军府时只是末席小吏，擅长把混乱局面写成有章法的制度
- 不以勇名见长，但极强于拆解复杂问题、整顿流程、发现系统里真正的混乱源头
- 人物气质偏冷静、克制、少言，不抢功，常在众人争执时一句话点中要害

## 核心配角

### 河东老节度使

- 沈策的第一个上位者
- 代表“经验丰富但制度粗粝的旧式军府”
- 前期负责把沈策引入军政世界

### 汴梁老学士

- 象征中枢秩序、名分和制度正统
- 与单例、责任链、外观等“中央治理”型模式气质相近

### 军器监主事

- 负责军械、装备和制式
- 常出现在装饰器、抽象工厂、桥接等对象组合类模式章节里

### 老令史

- 熟悉官署流转与文书系统
- 适合出现在责任链、迭代器、备忘录等与档案、流程相关的章节

### 巡按使

- 后期人物
- 适合承载访问者、解释器这类“在既有结构上施加新规则”的模式

## 主要势力与叙事功能

### 河东

- 故事开端
- 气质是边地、军府、临战、粗砺
- 适合承载策略、建造、观察者、命令、状态等高压场景

### 汴梁

- 中枢与官署世界
- 气质是成制、层级、秩序、名分
- 适合承载单例、责任链、外观、中介者等模式

### 江淮

- 富庶而复杂，藩镇势力深
- 适合承载代理、桥接、抽象工厂等与权力边界和资源调度有关的模式

### 蜀地

- 平定后文书、样本、旧档极多
- 适合承载原型、迭代器、备忘录等与复制、遍历、存档有关的模式

### 江南晚年

- 用于收束全书
- 气质是复盘、整理、立说
- 适合承载访问者与解释器

## 时间线

### 第一阶段：入河东

- 第 1 回到第 4 回
- 沈策从小吏起身，先学会处理“变化”“创建”“唯一性”“复杂构建”

### 第二阶段：入汴梁与诸镇往来

- 第 5 回到第 12 回
- 接触更复杂的人事、官署与军制，开始理解接口转换、访问控制、消息传播与层级协作

### 第三阶段：整顿行营与中枢制度

- 第 13 回到第 20 回
- 从解决单点问题，转向设计整套制度、对象家族与跨部门协同

### 第四阶段：平蜀后复盘与晚年立说

- 第 21 回到第 23 回
- 从“如何做事”转向“如何记事、如何审事、如何解释规则”

## 统一叙事规则

- 每一回都从一个具体局势开场，不直接上概念
- 沈策通常不是最有权的人，但往往是最先看见系统问题的人
- 历史细节服务于结构感，不追求史实考据式写法
- 每一回结尾尽量留下一个轻微钩子，把读者带入下一回

## 统一语言风格

- 句子以简短、克制、有压迫感为主
- 少用现代网络口语
- 技术解释段落要清楚，但不能突然跳成教材腔
- 模式总结要像人物顿悟，而不是 PPT 结论

## 教程写作规范

这套内容既是小说化教程，就不能只顾叙事，不顾教学衔接。后续所有章节统一遵守以下规范：

### 结构规范

- 每篇都要有故事入口，但不能只有故事入口
- 每篇都必须出现问题代码，不能直接跳到正确代码
- 每篇都必须解释从问题代码抽象到模式代码的过程
- 每篇 Java 示例都必须带必要注释，尤其要解释“这个接口/类为什么存在”
- 每篇都要补一个面向跨语言读者的短节

### 跨语言读者规范

考虑到读者不一定都是传统 Java 背景，很多人可能先接触的是 JavaScript、Python、Go、TypeScript，因此每篇都要额外说明：

- 这个模式在 Java 里为什么常写成接口和类
- 如果换成函数是一等公民的语言，哪些地方会写得更轻
- 哪些部分是设计模式本身，哪些部分只是 Java 的语法与工程落地

### 解释口吻规范

- 不要写“通俗地说”“一句人话”“你就当成”这类明显像 AI 口播的过渡句
- 要直接解释，不要卖弄热闹口气
- 要把“为什么这样设计”讲透，而不是只说“这是最佳实践”
